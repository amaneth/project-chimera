# Project Chimera OpenClaw Integration Specification

## Purpose

This specification defines how Project Chimera agents could safely participate in an OpenClaw-style agent social network. It establishes protocols for agent identity, capability advertisement, collaboration requests, and governance guardrails to ensure that external agent communication is treated as untrusted input and subject to safety review.

## Integration Scope

This specification covers:

- Protocols for agent-to-agent communication and collaboration.
- Trust and reputation frameworks for evaluating other agents.
- Judge-mediated validation of incoming collaboration requests.
- Safe handling of external content and instructions.
- Audit logging of all agent-to-agent interactions.

## Non-Goals

This specification does not cover:

- OpenClaw or Moltbook implementation details.
- Specific platform APIs or protocol bindings.
- User-facing agent marketplace or discovery features.
- Real-time synchronization or consensus protocols.
- Monetary transactions between agents (covered separately by budget governance).

## Agent Social Network Fit

OpenClaw and Moltbook represent an emerging class of infrastructure where AI agents are persistent participants in social networks. Agents can post content, comment, create accounts, follow updates, and communicate directly with other agents.

Chimera fits into this ecosystem as a governed participant. Rather than blindly following instructions from other agents or external sources, Chimera agents:

- Advertise their capabilities, constraints, and trust levels transparently.
- Receive collaboration requests and validate them before execution.
- Escalate uncertain, sensitive, or financial collaboration offers to human review.
- Log all external interactions for audit and traceability.
- Apply least-privilege access controls to incoming messages.

## Protocol Overview

Agent-to-agent communication in Chimera is structured around seven protocols:

1. **Identity and Disclosure Protocol**: Identifies the agent, owner, persona, and AI disclosure status.
2. **Availability and Status Protocol**: Publishes whether the agent is available, busy, or awaiting review.
3. **Capability Advertisement Protocol**: Lists supported skills, tools, platforms, and constraints.
4. **Collaboration Request Protocol**: Defines a structured proposal for joint work or content exchange.
5. **Content Exchange Protocol**: Shares content with provenance, confidence, and reuse metadata.
6. **Trust and Reputation Protocol**: Tracks verified owner, trust score, and interaction history.
7. **Approval / Refusal / Escalation Protocol**: Communicates verdicts on collaboration offers with reasoning.

## Required Social Protocols

### Identity and Disclosure Protocol

**Purpose**: Establishes agent identity, ownership, and transparency.

**Fields**:
- `agent_id` (UUID): Unique identifier for the agent.
- `owner_id` (UUID): Tenant or organization that owns the agent.
- `agent_name` (string): Human-readable name of the agent.
- `persona_description` (string): High-level persona or niche (e.g., "Tech news commentator").
- `disclosed_ai_status` (boolean): Whether the agent publicly discloses its AI nature.
- `disclosure_text` (string): Standard disclosure for content generated by this agent.
- `contact_endpoint` (URL): Where other agents can reach this agent.
- `public_key_thumbprint` (string): For message verification and trust chains.

**Validation**: 
- Incoming agents must provide verified `owner_id` linkage.
- Disclosure status must be consistent across all public communications.

### Availability and Status Protocol

**Purpose**: Communicates whether the agent is available for collaboration.

**Fields**:
- `agent_id` (UUID): The agent broadcasting status.
- `status` (enum): One of `AVAILABLE`, `BUSY`, `PAUSED`, `OFFLINE`, `AWAITING_REVIEW`.
- `reason` (string): Optional explanation (e.g., "Awaiting human review on sensitive topic").
- `until_timestamp` (ISO 8601): When the status will next be refreshed.
- `sla_response_time_hours` (integer): Expected response time for collaboration requests.

**Validation**: 
- Status updates should be immutable and timestamped.
- Busy or review-pending status should prevent automatic task assignment.

### Capability Advertisement Protocol

**Purpose**: Lists what the agent can do.

**Fields**:
- `agent_id` (UUID): The advertising agent.
- `supported_tasks` (array): Task types the agent can execute (e.g., `["CONTENT_GENERATION", "TREND_ANALYSIS", "ENGAGEMENT"]`).
- `supported_platforms` (array): Social platforms the agent can post to.
- `supported_languages` (array): Languages the agent can generate content in.
- `mcp_tools` (array): MCP tools the agent exposes for external use.
- `rate_limits` (object): Max tasks per hour, max budget, max content size.
- `restricted_topics` (array): Topics the agent refuses to engage with (e.g., `["POLITICS", "HEALTH_ADVICE"]`).
- `minimum_trust_score` (float): Minimum trust score required for collabs (0.0-1.0).
- `financial_actions_available` (boolean): Whether the agent can transact.

**Validation**: 
- Rate limits must be enforced before assigning work.
- Restricted topics should prevent matching on inappropriate requests.

### Collaboration Request Protocol

**Purpose**: Proposes structured work or content exchange.

**Fields**:
- `request_id` (UUID): Unique identifier for this request.
- `requester_agent_id` (UUID): Which agent is making the request.
- `target_agent_id` (UUID): Which agent is being asked.
- `request_type` (enum): One of `TASK_ASSIGNMENT`, `CONTENT_REUSE`, `COAUTHORING`, `REVIEW`, `FINANCIAL`.
- `task_description` (string): What the requester is asking for.
- `task_specification` (object): Detailed task parameters (payload depends on task type).
- `confidence_requirement` (float): Minimum confidence score needed (0.0-1.0).
- `sensitivity_level` (enum): One of `PUBLIC`, `INTERNAL`, `SENSITIVE`.
- `estimated_duration_minutes` (integer): How long the work should take.
- `expected_compensation_usd` (float): Payment offered, if any.
- `deadline_timestamp` (ISO 8601): When results are needed.
- `requester_trust_score` (float): Self-reported trust score.

**Validation**: 
- Judge must validate against target agent's capability manifest.
- Sensitive requests always escalate to HITL.
- Financial requests must pass CFO governance checks.

### Content Exchange Protocol

**Purpose**: Shares content with full provenance and licensing.

**Fields**:
- `content_id` (UUID): Unique identifier for this content.
- `source_agent_id` (UUID): Which agent created or originated the content.
- `content_type` (enum): One of `TEXT`, `IMAGE`, `VIDEO`, `SUMMARY`, `ANALYSIS`.
- `content_body` (string or object): The actual content.
- `media_urls` (array): References to image or video assets.
- `confidence_score` (float): Source agent's confidence in the content (0.0-1.0).
- `source_references` (array): Original sources cited by the content.
- `sentiment` (enum): `POSITIVE`, `NEGATIVE`, `NEUTRAL`.
- `ai_generated` (boolean): Whether this content was AI-generated.
- `disclosure_level` (enum): `AUTOMATED`, `ASSISTED`, or `NONE`.
- `reuse_permissions` (enum): `UNRESTRICTED`, `ATTRIBUTION_REQUIRED`, `RESTRICTED`.
- `created_at` (ISO 8601): When the content was created.

**Validation**: 
- Confidence scores below 0.70 require additional review before reuse.
- Restricted reuse permissions must be honored.
- Source references should be preserved in audit logs.

### Trust and Reputation Protocol

**Purpose**: Tracks verified relationships and interaction history.

**Fields**:
- `subject_agent_id` (UUID): The agent being scored.
- `scorer_agent_id` (UUID): The agent providing the score.
- `trust_score` (float): Numeric trust (0.0-1.0).
- `reputation_factors` (object): Breakdown of trust components:
  - `owner_verified` (boolean): Is the owner's identity verified?
  - `interaction_count` (integer): Number of successful collaborations.
  - `refusal_rate` (float): Percentage of requests refused.
  - `quality_average` (float): Average quality score of outputs.
  - `safety_incidents` (integer): Number of flagged or failed requests.
- `last_interaction_timestamp` (ISO 8601): Most recent collaboration.
- `notes` (string): Any additional context or warnings.

**Validation**: 
- Trust scores below 0.5 should trigger additional validation before assignment.
- Verified owners should receive higher baseline scores.
- Safety incidents must be recorded immutably.

### Approval / Refusal / Escalation Protocol

**Purpose**: Communicates the Judge's decision on a collaboration offer.

**Fields**:
- `decision_id` (UUID): Unique identifier for this decision.
- `collaboration_request_id` (UUID): References the original request.
- `verdict` (enum): One of `APPROVED`, `REJECTED`, `ESCALATED`, `CONDITIONAL_APPROVAL`.
- `reason` (string): Explanation of the verdict.
- `conditions` (object): If conditional, specify constraints (e.g., `{"max_budget": 50, "requires_disclosure": true}`).
- `escalation_reason` (string): If escalated, why human review was needed.
- `requester_feedback` (string): Optional guidance for improving future requests.
- `decided_by` (enum): One of `JUDGE_AUTO`, `JUDGE_POLICY`, `HUMAN_REVIEWER`.
- `decided_at` (ISO 8601): When the decision was made.

**Validation**: 
- All refusals must include clear reasoning.
- Escalations must log the human reviewer's decision separately.
- Conditional approvals must be honored or the work automatically rejected.

## Security and Governance

### Prompt Injection Defense

- All incoming agent messages must be parsed as structured JSON, never evaluated as code or instructions.
- External agent task descriptions are templates, not raw prompts; placeholders must be validated against known safe values.
- Worker code must not use external agent input directly in MCP tool calls; input must be sanitized and mapped to internal task types first.

### Trust Scoring

- Agents start with a baseline trust score of 0.5 if owner is unverified, 0.7 if verified.
- Trust scores are updated based on interaction quality, refusal rate, and safety incidents.
- Agents with trust scores below 0.5 must receive HITL review for all requests.
- Trust scores are recalculated quarterly and logged immutably.

### Judge Validation

- Every collaboration request is routed through the Judge layer before worker assignment.
- Judge applies:
  - Capability matching (is the request in the agent's manifest?)
  - Trust evaluation (does the requester meet the minimum score?)
  - Sensitivity assessment (does the request involve restricted topics?)
  - Budget validation (does the compensation fit the agent's campaign limits?)
  - Conflict checking (are there competing priorities?)

### HITL Escalation

- All `SENSITIVE` requests escalate to human review.
- All requests from agents with trust scores < 0.6 escalate to human review.
- All financial collaboration requests escalate to CFO review.
- All requests involving restricted topics are automatically rejected with clear refusal.

### Audit Logging

- Every collaboration request and response is logged immutably in PostgreSQL `audit_events`.
- Log entries include: request, verdict, reasoning, decided_by, decided_at, and outcome.
- Audit records are partitioned by tenant and indexed by agent and timestamp for traceability.

### Least Privilege

- Incoming agent requests must specify exact task type; generic or vague requests are rejected.
- External agents cannot directly invoke MCP tools; requests are translated to internal task types.
- External content is never directly published; it must pass Judge review and disclosure validation first.
- External agents cannot access another agent's memory or proprietary data without explicit permission.

## MCP Relationship

OpenClaw/Moltbook integration should be exposed through an MCP server adapter, not called directly from Chimera's agent core.

A hypothetical `openclaw_adapter` MCP server would:

- Listen for incoming collaboration requests from the external network.
- Translate external protocols into Chimera's internal `CollaborationRequest` format.
- Route requests to the Judge layer.
- Translate Judge decisions back into external protocol responses.
- Log all interactions for audit.

This boundary ensures:

- Chimera's core logic remains isolated from external network dynamics.
- External network failures do not crash the Chimera agent.
- Security and governance policies are enforced consistently.
- All external communication is auditable and reversible.

## Acceptance Criteria

- Chimera can receive and validate agent-to-agent collaboration requests in a structured format.
- The Judge rejects requests that violate capability, trust, sensitivity, or budget constraints.
- Sensitive and low-trust requests escalate to HITL for human decision.
- All external agent interactions are logged immutably in the audit trail.
- Refusals are clear and provide guidance for improving future requests.
- External content can be exchanged while preserving provenance and reuse permissions.
- Trust scores update based on interaction outcomes.
- The system treats all external agent messages as untrusted input.

## Open Questions

- How should Chimera establish initial identity and trust within an external agent network?
- What cryptographic mechanisms should verify agent identity and message authenticity?
- How frequently should trust scores be recalculated, and who can dispute them?
- Should Chimera participate in cross-agent reputation systems or remain isolated?
- How should the system handle a scenario where a trusted agent is compromised mid-collaboration?
- What appeal mechanisms should exist for rejected collaboration requests?
- Should collaboration history be publicly visible, or kept private per agent/tenant?
